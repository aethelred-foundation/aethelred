# AIP-0002: Proof-of-Useful-Work Consensus Specification

## Abstract

This AIP formally specifies the Proof-of-Useful-Work (PoUW) consensus mechanism used by Aethelred. Validators perform AI inference jobs inside Trusted Execution Environments (TEEs) and/or generate zero-knowledge ML proofs. Results are committed via ABCI++ vote extensions and aggregated to reach Byzantine fault-tolerant consensus on computation outputs, resulting in on-chain Digital Seals.

## Motivation

Traditional Proof-of-Work wastes energy on meaningless hashes. Proof-of-Stake secures economic value but produces no external utility. PoUW combines BFT security with verifiable AI computation — turning validator work into a public good while maintaining cryptographic finality guarantees.

## Specification

### 1. Job Submission

A compute job is submitted via `MsgSubmitJob`:
```protobuf
message MsgSubmitJob {
  string submitter = 1;
  bytes  model_hash = 2;   // SHA-256 of registered model
  bytes  input_hash = 3;   // SHA-256 of encrypted input data
  string verification_type = 4; // "tee" | "zkml" | "hybrid"
  string priority = 5;     // "standard" | "fast" | "urgent"
}
```

### 2. VRF Job Assignment

At each block, the scheduler assigns jobs to validators via a VRF (Verifiable Random Function) seeded by the previous block hash + validator cons key. This ensures:
- Uniform load distribution
- Unpredictability (no front-running)
- Verifiable on-chain (anyone can check assignment correctness)

### 3. Vote Extensions (ABCI++)

Each validator calls `ExtendVoteHandler` during the voting phase:
1. Fetches assigned jobs from `x/pouw` scheduler
2. Executes each job in a TEE and/or generates a zkML proof
3. Constructs a `VoteExtension` containing `ComputeVerification` results
4. Signs the extension with its Ed25519 consensus key
5. Returns the signed, size-bounded (≤5 MB) extension to CometBFT

### 4. Vote Extension Validation

`VerifyVoteExtensionHandler` validates each peer's extension:
- **Strict mode** (production): Signature required, extension hash required, simulated TEE rejected, BlockHeight+ChainID binding in UserData
- **Permissive mode** (devnet): Structural checks only

### 5. Consensus Aggregation

In `PrepareProposalHandler`, the block proposer:
1. Aggregates all vote extensions from `LocalLastCommit`
2. Groups verifications by `JobID` and `OutputHash`
3. Determines consensus when agreement power ≥ 67% (BFT threshold)
4. Injects `InjectedVoteExtensionTx` (type: `create_seal_from_consensus`) at the front of the block

### 6. Seal Creation

In `ProcessProposalHandler` / `DeliverTx`, the injected tx triggers `x/seal.CreateSeal`:
- Checks ≥ 67% power agreement
- Mints a Digital Seal NFT anchoring the output hash to block height
- Triggers fee distribution (40% validators / 30% treasury / 20% burn / 10% insurance)

### 7. Gas & Fee Model

- Base job fee: 0.001 AETH (governance adjustable)
- Dynamic multiplier: 1x–5x based on queue congestion (threshold: 70% utilization)
- Priority tiers: standard (1x), fast (2x), urgent (5x)

### 8. Reward Scaling

Validator rewards are scaled by reputation score [0, 100]:
```
reward = base_reward × (50 + reputation / 2) / 100
```
Range: 50% (rep=0) to 100% (rep=100) of base reward.

## Rationale

The hybrid TEE + zkML approach was chosen over:
- **TEE-only**: Susceptible to hardware supply chain attacks; no cryptographic proof
- **zkML-only**: Proof generation is too slow for per-block latency requirements
- **Hybrid**: TEE provides low-latency results; zkML provides cryptographic proof; consensus over both

## Backwards Compatibility

This AIP defines the initial PoUW mechanism. Future AIPs may extend it. The `VoteExtension.Version` field enables rolling upgrades.

## Security Considerations

- **Sybil resistance**: Minimum stake (1000 AETHEL) required for validator participation
- **TEE bypass**: Mitigated by zkML proof requirement in hybrid mode
- **Replay attacks**: Nonce + BlockHeight + ChainID binding in all attestations
- **Consensus liveness**: Panic recovery in all ABCI++ handlers; empty extensions accepted
- **Slashing**: Fraudulent attestations trigger 50% slash + permaban
