# AIP-0003: Digital Seal Standard

## Abstract

A Digital Seal is an on-chain attestation record minted by Aethelred when 2/3+ of validators reach consensus on a computation output. This AIP defines the Seal data structure, minting conditions, verification interface, and cross-chain relay format.

## Motivation

AI computation results need a portable, verifiable on-chain record that third parties - contracts, other chains, and off-chain systems - can trust without re-running the computation.

## Specification

### Seal Data Structure

```protobuf
message Seal {
  string id = 1;              // Globally unique: "{chain_id}/{height}/{job_id}"
  bytes  job_id = 2;          // SHA-256 of job parameters
  bytes  model_hash = 3;      // Registered model hash
  bytes  input_hash = 4;      // Encrypted input hash
  bytes  output_hash = 5;     // Consensus output hash (32 bytes, SHA-256)
  int64  block_height = 6;    // Block height at consensus
  string chain_id = 7;        // Chain identifier (replay protection)
  repeated ValidatorAttestation attestations = 8;
  int64  agreement_power = 9; // Voting power that agreed
  int64  total_power = 10;    // Total voting power in round
  string proof_type = 11;     // "tee" | "zkml" | "hybrid"
  google.protobuf.Timestamp created_at = 12;
}

message ValidatorAttestation {
  bytes validator_address = 1;
  AttestationType type = 2;
  bytes tee_quote = 3;         // Raw TEE attestation quote
  bytes zk_proof = 4;          // Serialized ZK proof
  bytes verifying_key_hash = 5;
}
```

### Minting Conditions

A Seal is minted when:
1. `agreement_power / total_power ≥ 0.67` (67% BFT threshold)
2. All attesting validators have valid job assignments for this block
3. The output hash is identical across all agreeing validators
4. No existing Seal exists for this `job_id`

### Verification Interface (EVM Precompile)

Address: `0x0000000000000000000000000000000000000500`

```solidity
interface ISealVerifier {
    function verifySeal(
        bytes32 jobId,
        bytes32 outputHash,
        uint64  blockHeight,
        bytes   calldata proof
    ) external view returns (bool valid, uint64 agreementPower, uint64 totalPower);
}
```

### Cross-Chain Relay (IBC)

Sealed results are relayed to other Cosmos chains via IBC packet:
```
Channel: "channel-0" (Aethelred → destination)
Port:    "aethelred.seal"
Packet data: { seal_id, output_hash, block_height, chain_id, agreement_bps }
```

## Rationale

The Seal is a minimally-sized on-chain record - it stores hashes, not raw data. Raw outputs are stored off-chain (IPFS / Arweave) with the `output_hash` as the content address.

## Backwards Compatibility

Seals are append-only. Once minted, a Seal cannot be modified or deleted. A new AIP is required to change the Seal data structure after the first mainnet Seal is minted.

## Security Considerations

- **Replay attacks**: Both `chain_id` and `block_height` are bound into each attestation
- **Seal forgery**: Requires compromising ≥ 33% of staked AETHEL (BFT assumption)
- **Output collision**: SHA-256 pre-image resistance; collision requires 2^128 operations
